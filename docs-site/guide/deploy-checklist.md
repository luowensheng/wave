# Production checklist

Before shipping a Wave server to real users, walk this list.

## Config hygiene

- [ ] `wave validate server.yaml` exits 0 (CI gate)
- [ ] `wave doctor server.yaml --json` exits 0 (CI gate against
      staging)
- [ ] `wave fmt server.yaml --check` exits 0 (pre-commit hook)
- [ ] Every secret is referenced via `${env:NAME}` or `${PLUGIN:...}`,
      never inline
- [ ] No `default: { host: 0.0.0.0 }` unless you mean it — prefer
      127.0.0.1 + reverse proxy or `--host 0.0.0.0` at CLI

## Security

- [ ] HTTPS is terminated (Fly/load balancer/reverse proxy)
- [ ] `cookie_secure: true` on every auth backend
- [ ] CSRF wrapper on every state-changing form route
- [ ] Webhook routes have `webhook_sig:` with the right provider
- [ ] Public routes have `cors_origins:` set (not `*` in prod for
      credentialed routes)
- [ ] `request_schema:` on routes accepting structured JSON bodies
- [ ] `limits[body_too_large]` reasonable for the route's max
      expected body (default 10 MB)
- [ ] Run `wave doctor` once with `BLOCK_PRIVATE_IPS=true` on any
      `dynamic_forward` route (SSRF guard)

## Reliability

- [ ] `limits[rate_limited]` on every public-facing route
- [ ] `limits[circuit_open]` on routes that call downstream
      services (forward / api / plugin)
- [ ] `outbox_db:` set if you send any external webhooks/emails
- [ ] `wave outbox list --db ./outbox.db` is monitored (alert when
      `pending > N` or `dlq > 0`)
- [ ] Health probes wired to `/healthz` (liveness) and `/readyz`
      (readiness)
- [ ] Migrations run via `wave migrate up` as a deploy step (not
      at server boot)

## Observability

- [ ] Prometheus scraping `/metrics`
- [ ] Dashboards for: p99 latency by route, error rate by status,
      rate-limit rejects/min, SSE subscriber count
- [ ] Alerts on: `/readyz` failing, outbox DLQ > 0, p99 latency
      breaching SLO, error rate > 1%
- [ ] OTLP traces shipping to Tempo / Jaeger / Honeycomb
- [ ] Structured logs shipping to a SIEM / Loki / Datadog
- [ ] Audit log on every mutation (see
      [audit log recipe](/cookbook/audit-log))

## Capacity

- [ ] Load-tested at 2× peak expected RPS with realistic payloads
- [ ] Body-size limits are realistic (10 MB default may be too
      tight for file uploads, too generous for JSON APIs)
- [ ] SQLite WAL mode enabled (it is by default in Wave's sqlite
      backend — verify with `PRAGMA journal_mode`)
- [ ] Plugin processes scaled to expected concurrency (subprocess
      kind spawns one process per request — high RPS needs
      `longlived`)

## Deploy pipeline

- [ ] Image tag is a SemVer release (`:v0.1.0`), not `:latest`
- [ ] Release notes published (auto-generated by goreleaser)
- [ ] Cosign verification step in deploy:

      ```sh
      cosign verify ghcr.io/luowensheng/wave:v0.1.0 \
        --certificate-identity-regexp='https://github.com/luowensheng/wave/' \
        --certificate-oidc-issuer='https://token.actions.githubusercontent.com'
      ```

- [ ] SBOM attached to the release (auto, via syft)
- [ ] Rollback plan documented (previous image tag pinned, DB
      migration is reversible or backed up first)

## Backup & recovery

- [ ] SQLite file is on a snapshotted volume OR has scheduled
      `litestream replicate` to S3
- [ ] Outbox DB is on the same backed-up volume
- [ ] Restore drill performed at least once
- [ ] Plugin binaries are versioned (kept in your container or
      mounted from a known path)

## Documentation

- [ ] Runbook for the on-call (link to `/admin`, `/metrics`, log
      search query, common incidents)
- [ ] CHANGELOG entry for the version you're shipping
- [ ] Architecture doc current (every Wave route type and plugin
      noted)

## Wave-specific reminders

- [ ] `methods:` (plural) wherever `cors_origins:` is set (avoid
      the OPTIONS-405 footgun)
- [ ] `if_empty_status: 404` on GET-by-id storage-access routes
- [ ] `LIMIT 1` on single-row lookups for clean `.Data.col` access
- [ ] Pre-1.0: read the CHANGELOG before any version bump

When every checkbox is green, ship it.
