## Becoming a maintainer

There is no formal process yet — the project is young. The current
shortcut: land 3-5 substantive PRs that the existing maintainer
merged, then nominate yourself by opening an issue. As the project
grows we'll formalize the rules.

## Maintainer responsibilities

- **Triage** issues within 24 hours on weekdays (initial response;
  resolution can take longer).
- **Review** pull requests within 3 days. If you can't, leave a comment
  saying when you can — silence is worse than delay.
- **Hold the bar** on code style (see CLAUDE.md), test coverage, and
  documentation. A PR is not merged until those are in place.
- **Communicate** in Discussions / Discord — make decisions visible.
- **Cut releases** following the process in CONTRIBUTING.md → "Releasing".

## Decision making

For now: lazy consensus. Anything a maintainer doesn't object to
within 72 hours is approved. For breaking changes, open an issue
labeled `discuss` and wait for an explicit thumbs-up from at least
one other maintainer before merging.

## Stepping down

Maintainers who become inactive for 6+ months and don't respond to
a check-in get moved to "Emeritus" in this file. No drama, no shame
— life happens. You can always come back.
